## 02 — Market Signals

**Signal 1: AI version provenance untracked across active program portfolios**
NGOs operating multiple programs with different AI tool versions simultaneously have no mechanism for tracking which version produced which program's decision basis. Version provenance is not a documentation preference — it is the difference between a defensible decision record and a reconstruction gap that no subsequent documentation can close.

**Signal 2: Donor AI questionnaires arriving as organizational self-discovery instruments**
Institutional donor AI governance questionnaires are functioning as self-discovery tools for NGOs: the process of answering reveals governance configurations NGOs did not know they had. The revelation function of questionnaires is separate from the accountability function. Organizations are learning about their own AI deployments by attempting to respond.

**Signal 3: Beneficiary nFADP rights activation for AI-inferred conclusions arriving before triage architecture**
AI-inferred conclusions about beneficiary vulnerability, eligibility, and need are activating nFADP rights the organization has no triage architecture to discharge. The data subject request is not the problem — it is the end of a process the organization did not design. No Geneva NGO has a triage architecture that determines which requests require full AI inference disclosure and which require different response pathways.

**Signal 4: Geneva AI governance framework ecosystem provenance gap**
The Geneva AI governance landscape includes UNAIWIG, ICRC, OCHA, donor-specific AI frameworks, and Swiss regulatory frameworks — none of which are individually identified in an authoritative provenance map. Organizations following multiple frameworks simultaneously cannot determine which framework's standards apply to which operational moment. The ecosystem is dense; the map is absent.

**Signal 5: Program-close AI decision inheritance checklists absent from all standard program-close frameworks**
Program-close checklists cover beneficiary data disposal, financial reconciliation, and donor reporting. No program-close checklist in common Geneva NGO use covers AI-influenced decisions as a named category. Decisions made through AI systems continue to produce consequences after program termination that the closing process does not address.

**Signal 6: AI tool update evaluation as permanently unbudgeted governance cost**
AI tool vendors update their systems continuously. Each update requires evaluation: did the update change outputs in ways material to program decisions? This evaluation is governance work. It has no budget category, no staff designation, and no defined scope. At NGO scale with limited IT capacity, the cumulative burden of continuous update evaluation is structurally invisible and ungoverned.

## 03 — Competitive Analysis

**Gap 1: AI version provenance tracking architecture for NGO program portfolios**
No service offers AI version provenance tracking for organizations running multiple programs simultaneously. The architecture would need to track which AI version was active for which program, which decisions were made under which version, and what the implications are for program defensibility. This is distinct from model deprecation management (which addresses what to do when a model is retired) and from tool migration documentation (which addresses the act of switching vendors). This is about maintaining version awareness across an active portfolio.

**Gap 2: Donor AI governance questionnaire design review as a discrete service**
No service offers review of donor AI governance questionnaires before submission. The review would assess whether the questionnaire actually measures what the donor intends, whether the NGO's response will be interpreted correctly, and whether the NGO's genuine governance adequacy will be obscured by questionnaire design gaps. This is distinct from questionnaire response preparation (which assumes the questionnaire is well-designed) and from compliance review (which evaluates against known standards).

**Gap 3: Beneficiary nFADP data subject request triage architecture for AI-inferred data**
No architecture exists for triaging beneficiary nFADP data subject requests involving AI-inferred content at NGO scale. The architecture would need to determine: what data was inferred, what conclusions were drawn, which conclusions are active in program decisions, what the beneficiary is entitled to know, what the organization can disclose without exposing methodology it is not prepared to explain, and what the response pathway is for each category of inference. This is distinct from general nFADP data subject request processes (which address directly collected data) and from consent architecture design (which addresses future data collection).

**Gap 4: Geneva AI governance framework provenance mapping service**
No publication has produced an authoritative map of Geneva AI governance framework provenance: which framework applies to which operational moment, which frameworks overlap, which are hierarchically superior, and how to determine which standard is operative when multiple frameworks apply simultaneously. Organizations following the ecosystem without the map are making framework selection decisions without knowing what they are choosing.

**Gap 5: Post-program AI decision inheritance management as program-close service category**
No program-close service addresses AI-influenced decisions as a named inheritance category. The service would identify which AI-influenced decisions persist after program termination, what downstream obligations the organization retains, what documentation should travel with the decisions, and how to design program-close AI accountability protocols. This is distinct from data disposal (which addresses records not decisions) and from beneficiary consent architecture (which addresses enrollment not termination).

**Gap 6: AI tool update evaluation as discrete governance budget category for NGOs**
No Geneva NGO has a budget category for AI tool update evaluation. The service would establish what the evaluation function costs, how to staff it, what the evaluation scope is per update, and how to distinguish between updates that require full re-evaluation and updates that do not. This is distinct from IT operational budgeting (which addresses technology infrastructure costs) and from compliance monitoring (which addresses regulatory obligation discharge).

## 04 — Compliance Context

**ZEWO certification coverage and the AI governance gap it does not address:**
ZEWO certification verifies financial governance, fundraising ethics, and organizational governance in those domains. It does not include AI governance as a certification domain. NGOs holding valid ZEWO certification are not certified for AI governance adequacy. Institutional donors interpreting ZEWO certification as a proxy for overall organizational governance quality are applying a standard the certification does not support. The gap between what ZEWO certifies and what institutional donors assume is a structural miscommunication in the Geneva NGO accountability ecosystem.

**Institutional donor AI conditions in grant agreements creating obligations without interpretive guidance:**
Institutional donors are inserting AI governance conditions into grant agreements at increasing frequency. The conditions require NGOs to maintain AI governance standards without defining what those standards are. The interpretation burden falls on the NGO at the moment of compliance verification, often during a program audit when the interpretation decision has ongoing consequences. No Geneva donor has published interpretive guidance for its own AI governance conditions.

**nFADP Articles 13 and 21 obligations for AI-inferred beneficiary conclusions:**
Art. 13 requires explanation of automated decisions affecting data subjects. Art. 21 requires documentation of AI-assisted decision logic. When NGOs use AI to draw conclusions about beneficiaries — vulnerability levels, eligibility scores, needs assessments — those conclusions are automated decisions subject to Art. 13. The obligation exists regardless of whether the organization designed its systems to discharge it. The practical challenge for NGOs is that AI-inferred conclusions often represent the output of proprietary model logic the organization does not fully understand, making the explanation obligation structurally difficult to satisfy.

**Art. 75 ZGB board member personal liability for AI governance oversight:**
Board members who approve AI strategies without adequate review mechanisms face personal liability under Art. 75 ZGB. The liability attaches to oversight failures, not to AI outcomes. A board that approved an AI system two years ago and has not reviewed its current behavior may be accumulating personal liability continuously. The Art. 75 exposure is present-tense and compounding.

**FADP and Swiss hosting requirements for Geneva NGO AI operations:**
NGOs operating AI tools through vendor infrastructure are processing beneficiary data in systems subject to Swiss data protection law. The FADP (and nFADP for ongoing processing) applies to the organization's data processing regardless of where the AI vendor is headquartered. NGOs using US-based AI vendors are still subject to Swiss data protection obligations for that processing. The assumption that vendor location resolves the compliance question is incorrect.

**Geneva humanitarian AI standards ecosystem without published Swiss law compatibility mapping:**
The Geneva humanitarian AI standards ecosystem — UNAIWIG principles, ICRC AI ethics handbook, OCHA guidance, donor-specific frameworks — operates as an independent accountability system. Organizations following Geneva humanitarian standards may be in compliance with those standards while simultaneously in an unresolved position under Swiss data protection law. No authoritative mapping exists between Geneva humanitarian AI standards and Swiss legal obligations. The standards ecosystem is substantial; the mapping is absent.

## 06 — Findings Summary

**Finding 1: AI version provenance untracking as discrete reconstruction gap across active NGO program portfolios**
When NGOs cannot identify which AI version produced which program's decision basis, they cannot reconstruct the analytical conditions under which program decisions were made. The gap is not about documentation quality — it is about the absence of version tracking architecture that the organization never built. A program history assembled from documentation created without version tracking is inconsistent in ways no subsequent documentation program can repair. The problem is present-tense for any organization operating multiple AI-assisted programs simultaneously and becomes acute at program audit or donor inquiry. Distinct from all prior Phase 04 model deprecation findings (which addressed model retirement events) and from C276 Finding 4 (which named mid-program deprecation as a specific event without isolating the ongoing version provenance tracking gap across active portfolios as its own structural problem requiring continuous architecture, not a one-time deprecation response protocol).

**Finding 2: Donor AI governance questionnaire design as a discrete organizational self-discovery instrument with distorting effects on genuine governance adequacy**
The institutional donor AI governance questionnaire is functioning as an organizational self-discovery tool: the process of responding reveals configurations NGOs did not know they had. The questionnaire's design — what it asks, how it frames questions, what evidence formats it requests — does not necessarily correspond to what the donor actually intends to measure. Organizations with genuine AI governance adequacy may produce responses that appear inadequate because the questionnaire does not capture what they actually do. Organizations with genuine governance gaps may produce responses that appear adequate because the questionnaire does not surface the gaps. The questionnaire functions as a measurement instrument with unexamined validity. No Geneva service offers pre-submission questionnaire design review as a discrete offering. Distinct from all prior Phase 04 donor AI questionnaire findings which addressed questionnaire response preparation and documentation adequacy without identifying questionnaire design quality as a specific mechanism creating systematic misrepresentation of genuine governance adequacy independent of the organization's actual practices.

**Finding 3: Beneficiary nFADP data subject request triage architecture as a structurally absent response infrastructure for AI-inferred content**
AI-inferred beneficiary conclusions activate nFADP Art. 13 and Art. 21 obligations. The response requires the organization to determine: what was inferred, what the inference logic is, what conclusions are active in program decisions, what the beneficiary is entitled to know, what the organization can disclose without exposing proprietary AI methodology it is not prepared to explain, and what the appropriate response pathway is for each inference category. Most Geneva NGOs have no triage architecture for this determination. The absence is not a documentation gap — it is a response infrastructure gap: the organization does not have a process for making these determinations at the moment requests arrive. The problem is structurally different from responding to requests for directly collected data because AI-inferred content requires the organization to understand its own AI system's logic before it can determine what to disclose. Distinct from C271 Finding 4 (which named nFADP data subject request response infrastructure absence as a general condition) by isolating the specific triage architecture problem: the organization needs to make decisions about disclosure scope, methodology exposure, and response pathway selection before it can produce a response. This is a triaging function, not a documentation function.

**Finding 4: Geneva AI governance framework ecosystem provenance mapping as a structural accountability absence distinct from framework content analysis**
The Geneva AI governance framework ecosystem includes UNAIWIG principles, ICRC AI ethics operationalization handbook, OCHA AI guidance, institutional donor-specific AI frameworks, and Swiss regulatory frameworks. Each is developed by a different actor with different accountability relationships, different geographic scope, and different enforcement mechanisms. No authoritative provenance map exists: which framework is hierarchically superior, which applies in which operational moments, which create legally operative obligations versus aspirational guidance, and how NGOs should determine which framework governs which of their decisions. Organizations following multiple frameworks simultaneously cannot identify which standard is operative when frameworks conflict. The provenance map is not a content analysis exercise — it is an accountability architecture question: who is accountable to whom for following which standard, and through what mechanism. Distinct from all prior Phase 04 Geneva humanitarian AI standards findings (which named the framework ecosystem's existence without identifying the specific provenance mapping absence as a discrete accountability architecture problem requiring its own solution architecture).

**Finding 5: Post-program AI decision inheritance as a named program-close accountability domain structurally distinct from beneficiary data disposal**
Program-close checklists in Geneva NGOs cover beneficiary data disposal, financial reconciliation, and donor reporting. AI-influenced decisions made during the program do not appear in any standard program-close checklist as a named category. The decisions continue to produce consequences after program termination: referrals made on the basis of AI-influenced categorizations, downstream organizations acting on AI-influenced assessments, beneficiary records containing AI-inferred conclusions that may affect future service eligibility. The inheritance is structural — it exists regardless of whether the closing process addresses it. Data disposal frameworks do not cover it because data disposal addresses records, not decisions. The accountability domain requires its own named category and its own program-close protocol. Distinct from C270 Finding 1 (which named post-program AI decision continuity as a named accountability domain) by focusing specifically on the program-close checklist absence: the problem is not that decisions continue to have consequences, it is that no standard program-close framework in Geneva NGO use includes AI decision inheritance as a discrete closing category requiring documentation and handover protocols.

**Finding 6: AI tool update evaluation as a permanently unbudgeted governance cost creating structural under-governance at NGO scale**
AI tool vendors update their systems continuously. Each update requires evaluation: did the change affect outputs in ways material to program decisions? This evaluation is governance work. It has no budget category in standard NGO organizational budgets, no designated staff responsibility, and no defined scope. At NGO scale with limited IT capacity, the cumulative burden of continuous update evaluation is structurally invisible — it does not appear in governance reports, it is not categorized as a governance cost, and it competes with program delivery for staff time that has no governance budget designation. The cost is real; the budget architecture to account for it does not exist. Organizations are under-governed in AI update evaluation not because they choose to be but because the governance cost has no budget home. Distinct from C273 Finding 1 (which named program-team AI governance absorption as a structural governance failure) by isolating the specific mechanism: the absorption happens because update evaluation has no budget category and no defined scope, making it structurally impossible to resource properly. The governance gap is created by budget architecture absence, not by organizational priority failure.
